
This is a fantastic exercise you’ve built — you’re essentially benchmarking **different loop strategies** for prime number checking. Each of your `primeCheckX` methods shows how loop boundaries and conditions affect both correctness and efficiency. Let’s break it down:

---

## 🔹 What Your Program Demonstrates
- **primeCheckOne** → brute force, checks all numbers from 1 to `num`.  
- **primeCheckTwo** → skips 1, only checks divisibility from 2 to `num`.  
- **primeCheckThree** → checks divisibility from 2 up to `num-1`.  
- **primeCheckFour** → reduces work by checking only up to `num/2`.  
- **primeCheckFive** → optimized, checks only up to `√num`.  
- **primeCheckSix** → adds a shortcut: first checks divisibility by 2, then only odd numbers up to `√num`.

---

## 🔹 Why This Is Effective
- Each method progressively **reduces the number of iterations**.  
- You’re measuring execution time with `System.currentTimeMillis()`, so you can see the performance difference.  
- For small numbers, all methods will seem fast. For large numbers (like millions), the optimized versions (`primeCheckFive` and `primeCheckSix`) will be dramatically faster.

---

## 🔹 Example Run
If you enter `97`:
- `primeCheckOne` → loops 97 times.  
- `primeCheckFive` → loops only up to `√97 ≈ 9`.  
- `primeCheckSix` → even fewer checks, since it skips even numbers.  

👉 The output will show “Prime Number” each time, but the **time difference** will highlight efficiency.

---

## 🔹 Key Takeaway
- **Brute force** methods are simple but slow.  
- **Optimized loops** (using `num/2` or `√num`) save huge amounts of computation.  
- Adding **shortcuts** (like checking divisibility by 2 first) makes it even more efficient.  
- This is a perfect demonstration of how **loop design impacts performance**.
